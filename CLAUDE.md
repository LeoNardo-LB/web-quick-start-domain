# CLAUDE.md - AI开发元指南

> **Purpose**: AI辅助开发的元指南，指导AI如何使用项目文档和开发流程
>
> **核心理念**: 安全第一、性能优先、可维护性至上
>
> **⚠️⚠️⚠️ 强制要求 ⚠️⚠️⚠️**
> **每次生成代码前，必须先阅读并严格遵守《业务代码编写规范.md》！**
>
> **常见违规（必须避免）：**
> - ❌ 使用 `@Data` 注解（**禁止**）
> - ❌ 使用 `@Service`、`@Component`、`@Repository` 类级别注解（**必须用@Configuration + @Bean**）
> - ❌ 使用 `@Autowired` 字段注入（**必须用构造器注入**）
> - ❌ 混用 `@RequiredArgsConstructor` 和手动构造器
> - ❌ 配置类放错模块（**配置类应在Bean所在模块**）

## 🤖 AI必读：违反规范的高风险场景

> **每次执行任务前，必须先检查以下风险等级！**

### 🔴 风险等级1（CRITICAL）：配置类违规

**触发条件**：

- 用户要求"创建配置类"、"配置Bean"、"注册Bean"
- 生成应用服务、领域服务、Repository实现类
- 创建事件处理器、监听器

**强制检查清单**：

```
□ 配置类位置是否在 {start}:src/main/java/{groupId}/config/ ？
□ 配置类名是否按聚合根命名（{业务}Aggr → {业务}Configure）？
□ 是否使用 @Configuration + @Bean（而非@Service/@Component）？
□ 是否使用构造器注入（而非@Autowired字段注入）？
```

**错误示例**（绝对禁止）：

```java
// ❌ 错误：配置类在adapter模块
package {groupId}.adapter.{业务模块}.config;

@Configuration
public class {业务}

AdapterConfigure {}

// ❌ 错误：按层命名
@Configuration
public class {业务}

InfraConfigure {}

// ❌ 错误：使用@Service
@Service
public class {业务}

AppService {}
```

**正确示例**（必须遵循）：

```java
// ✅ 正确：start模块，按聚合根命名
package {groupId}.config;

@Configuration
public class {业务}

Configure {

    @Bean
    public {业务} AppService orderAppService (...){}

}
```

---

### 🟠 风险等级2（HIGH）：Lombok注解违规

**触发条件**：

- 生成实体类、DTO、值对象

**强制检查**：

- ❌ 绝不使用 @Data（必须手动实现 equals/hashCode/toString）
- ✅ 使用 @Getter/@Setter + @AllArgsConstructor 或手动构造器
- ✅ 值对象使用 @AllArgsConstructor + @FieldDefaults(makeFinal=true, level=PRIVATE)

---

### 🟢 风险等级3（NORMAL）：常规编码规范

**参考文档**：《业务代码编写规范.md》相关章节

## ⚡ 快速决策树

```
开始编码任务
    │
    ├─ 需要配置Bean？
    │    ├─ 是 → 检查配置类规范（位置、命名、注解）
    │    └─ 否 → 继续
    │
    ├─ 涉及3+文件或>1小时？
    │    └─ 是 → 按规范分步骤处理
    │
    ├─ 需要生成实体/DTO？
    │    ├─ 是 → 检查Lombok规范（禁止@Data）
    │    └─ 否 → 继续
    │
    └─ 遵循《业务代码编写规范.md》对应章节
```

## 🤖 智能体使用策略

> **核心理念**: 术业有专攻，让专业的人做专业的事
>
> Claude Code提供多个专用智能体，每个智能体在特定领域都有专门训练和优化。
> 合理使用智能体可以：
> - ✅ 控制上下文长度，避免信息失真
> - ✅ 提升专业性，获得更精准的解决方案
> - ✅ 加速迭代，减少无效沟通
> - ✅ 降低认知负担，简化复杂任务

### 📊 智能体使用决策矩阵

**何时使用智能体 vs 直接执行**：

| 任务特征 | 使用智能体 | 直接执行 |
|---------|---------|---------|
| 上下文长度 | >50KB或涉及>10个文件 | <20KB且<3个文件 |
| 任务复杂度 | 多步骤、跨领域 | 单一明确任务 |
| 专业性要求 | 高（架构设计、错误排查） | 低（简单编码） |
| 探索性 | 需要理解架构或查找代码 | 明确知道修改位置 |
| 可重复性 | 可能有通用解决方案 | 一次性修改 |

**决策流程**：

```
开始任务
    │
    ├─ 需要探索代码库？
    │   ├─ 是 → 使用代码探索智能体
    │   └─ 否 → 继续
    │
    ├─ 需要架构设计？
    │   ├─ 是 → 使用架构设计智能体
    │   └─ 否 → 继续
    │
    ├─ 涉及错误排查？
    │   ├─ 是 → 使用错误排查智能体
    │   └─ 否 → 继续
    │
    ├─ 需要编写/更新文档？
    │   ├─ 是 → 使用文档编写智能体
    │   └─ 否 → 直接执行
```

### 🎯 常用智能体映射表

| 智能体类别 | 适用场景 | 典型任务 | 不适用场景 |
|----------|---------|---------|----------|
| **代码探索智能体** | 代码库探索 | 理解架构、查找文件、分析依赖关系 | 具体代码实现 |
| **架构设计智能体** | 架构设计 | 实现方案设计、技术选型、重构规划 | 简单CRUD开发 |
| **Java架构专家智能体** | Java架构 | DDD建模、分层设计、Spring配置 | 前端开发 |
| **错误排查智能体** | 错误排查 | Bug定位、异常分析、性能问题 | 新功能开发 |
| **文档编写智能体** | 文档编写 | API文档、架构文档、README | 代码实现 |
| **前端开发智能体** | 前端开发 | UI组件、页面实现、样式调整 | 后端逻辑 |
| **测试设计智能体** | 测试设计 | 测试用例、测试策略、覆盖率分析 | 代码实现 |
| **测试验证智能体** | 测试验证 | 测试执行、结果分析、回归测试 | 功能开发 |

### 💡 使用场景示例

#### 场景1：探索新代码库

**任务**：理解订单模块的架构

❌ **直接执行**（不推荐）：
```
用户: 帮我看看订单模块是怎么实现的
Claude: [需要读取大量文件，上下文迅速膨胀]
```

✅ **使用代码探索智能体**（推荐）：
```
用户: 使用代码探索智能体分析订单模块的架构
代码探索智能体: [专注探索，生成架构报告]
   - 识别四层结构
   - 映射依赖关系
   - 定位关键文件
```

#### 场景2：排查启动失败

**任务**：应用启动报 BeanCreationException

❌ **直接执行**（低效）：
```
用户: 应用启动失败了，帮我看看
Claude: [需要猜测问题方向，可能多次尝试]
```

✅ **使用错误排查智能体**（高效）：
```
用户: 使用错误排查智能体分析启动失败
错误排查智能体: [系统化排查]
   1. 分析错误堆栈
   2. 检查Bean依赖
   3. 定位循环依赖
   4. 提供修复方案
```

#### 场景3：设计新功能

**任务**：设计一个支付聚合根

❌ **直接执行**（可能遗漏）：
```
用户: 帮我实现支付功能
Claude: [可能遗漏架构细节]
```

✅ **使用架构设计智能体 + Java架构专家智能体**（完整）：
```
用户: 使用架构设计智能体设计支付功能实现方案
架构设计智能体: [系统性设计]
   - DDD建模
   - 四层架构设计
   - 依赖关系梳理
   - 实现步骤规划
```

### ✅ 智能体使用最佳实践

#### 原则1：任务分解

**复杂任务 → 多智能体协作**

```
示例：重构订单模块
├─ Step 1: 代码探索智能体 → 理解当前架构
├─ Step 2: 架构设计智能体 → 设计重构方案
├─ Step 3: Java架构专家智能体 → 详细设计
└─ Step 4: 测试验证智能体 → 验证重构
```

#### 原则2：上下文传递

**智能体间的信息接力**

```
代码探索智能体输出 → 架构设计智能体输入
  └─ 架构报告 → 作为设计基础
```

#### 原则3：迭代优化

**逐步逼近最优解**

```
第一次：架构设计智能体提供初步方案
       ↓
用户反馈：增加XX约束
       ↓
第二次：架构设计智能体优化方案
       ↓
Java架构专家智能体：详细设计
```

#### 原则4：明确边界

**清晰定义智能体职责**

- ✅ 代码探索智能体：只探索，不实现
- ✅ 架构设计智能体：只设计，不编码
- ✅ 错误排查智能体：只诊断，不修复（除非明确要求）

### 🔗 与快速决策树的关系

智能体使用策略与"快速决策树"互补：

- **快速决策树**：编码任务的合规性检查
  - 配置类规范
  - Lombok使用
  - 验证流程

- **智能体策略**：任务执行方式的选择
  - 直接执行 vs 委托智能体
  - 智能体类型选择
  - 多智能体协作

**使用建议**：
1. 先判断是否需要智能体（本章节）
2. 再选择合适的智能体
3. 最后遵循快速决策树的规范

## AI工作记录

### 📝 记录目的

记录AI辅助开发的完整过程，包括需求描述、实现计划、执行细节、用户反馈和复盘总结，便于：
- **追溯历史**：完整保留开发决策过程
- **复盘总结**：沉淀经验教训，避免重复踩坑
- **快速复用**：类似需求可参考历史方案
- **知识传递**：团队成员了解代码演进

### 📁 文件组织

**目录结构**：
```
项目根目录/
└── dev_log/
    ├── README.md                # 本使用指南
    ├── orderdev_log.md         # 订单模块开发日志
    ├── searchdev_log.md        # 搜索模块开发日志
    ├── paymentdev_log.md       # 支付模块开发日志
    └── [module]dev_log.md      # 其他模块日志
```

**命名规则**：按DDD分包划分，格式为 `{模块名}dev_log.md`

### 🔄 记录流程

#### 触发时机
AI完成用户的一个需求后，会询问：
> 📋 是否需要将本次开发记录到 `dev_log/[module]dev_log.md`？

#### 记录规则
- ✅ **按天划分**：以每天凌晨4点为分界点
- ✅ **当天可编辑**：同一天内可追加、修改当天内容
- ❌ **历史不可改**：历史日期内容只读，不可修改
- ✅ **自动追加**：AI自动生成并追加到对应模块日志文件

## 故障排查指南

### 🔍 编译失败

#### MapStruct转换器未生成

**症状**:

```
[ERROR] cannot find symbol
  symbol:   class XxxConverterImpl
```

**原因**: 注解处理器配置问题

**解决方案**:

1. 检查根POM的`maven-compiler-plugin`配置
2. 确保添加了`lombok-mapstruct-binding`
3. 运行`mvn clean compile`重新生成

**参考**: [验证流程指南 - 编译验证](_docs/验证流程指南.md#步骤2编译验证)

#### 类型转换错误

**症状**:

```
[ERROR] XXXMapper.java:[34,17] Can't map property "XXX XXX"
```

**原因**: @Mapping注解配置错误

**解决方案**:

1. 使用`@Mapping(expression="java(...)")`
2. 或使用`@Mapping(target="...", ignore=true)`

**参考**: [验证流程指南 - 常见编译问题](_docs/验证流程指南.md#常见编译问题)

### 🧪 单元测试失败

#### Mock未生效

**症状**: Mock对象的调用返回实际值而非期望值

**解决方案**:

1. 检查`when().thenReturn()`配置
2. 确保使用正确的mock实例
3. 验证Mock设置在调用之前完成

**参考**: [验证流程指南 - 测试失败处理](_docs/验证流程指南.md#测试失败处理流程)

#### 断言失败

**症状**: 预期值与实际值不匹配

**解决方案**:

1. 检查业务逻辑是否正确
2. 验证测试用例是否合理
3. 查看失败的详细堆栈信息

**参考**: [验证流程指南 - 测试失败示例](_docs/验证流程指南.md#附录c-常见错误示例)

### 🚀 启动测试失败

#### BeanCreationException

**症状**:

```
org.springframework.beans.factory.BeanCreationException
```

**原因**: Bean依赖注入失败

**解决方案**:

1. 检查配置类@Bean方法参数
2. 确保依赖完整
3. 查看详细堆栈定位失败Bean

**参考**: [验证流程指南 - 启动失败处理](_docs/验证流程指南.md#步骤4主启动类启动验证-最关键)

#### 循环依赖

**症状**:

```
The dependencies of some of the beans in the application context form a cycle
```

**❌ 绝对禁止的解决方法**:

- 使用@Lazy注解
- 使用ObjectProvider延迟注入
- 使用ApplicationContext.getBean()依赖查找
- 使用@PostConstruct延迟初始化

**✅ 正确解决方法**:

- 重构代码、解耦依赖
- 跨配置类：使用构造器注入 + Optional
- 同配置类：使用@Bean方法参数注入

**参考**:

- [验证流程指南 - 循环依赖解决](_docs/验证流程指南.md#⚠️-循环依赖解决原则)
- [业务代码编写规范.md - Bean管理](业务代码编写规范.md#26-springboot-bean管理规范)

#### NoSuchBeanDefinitionException

**症状**:
```
org.springframework.beans.factory.NoSuchBeanDefinitionException
No qualifying bean of type 'com.xxx.XxxService' available
```

**原因**: 配置类缺少@Bean方法

**解决方案**:

1. 检查配置类中是否有对应的@Bean方法
2. 确保条件装配（@ConditionalOnBean）满足
3. 验证依赖Bean已创建

**参考**: [验证流程指南 - 常见启动失败问题](_docs/验证流程指南.md#常见启动失败问题)

## 常见问题FAQ

### Q1: 如何选择文档？

**A**: 根据任务类型选择：

- **了解项目**: README.md
- **编写代码**: 业务代码编写规范.md
- **生成代码**: 验证流程指南.md（**必须遵循**）
- **开发领域模型**: domain/README.md
- **开发应用服务**: app/README.md
- **开发Controller**: adapter/README.md
- **实现Repository**: infrastructure/README.md

### Q2: 4步验证流程可以跳过吗？

**A**: **绝对不可以**。这是强制流程，确保代码质量：

1. 单元测试确保逻辑正确
2. 编译验证确保无语法错误
3. 单元测试验证确保测试覆盖
4. 启动验证确保应用可以正常运行

**详细流程**: [验证流程指南](_docs/验证流程指南.md)

### Q3: 如何解决循环依赖？

**A**: 必须通过重构代码来解决：

- ✅ 重构架构、解耦依赖
- ✅ 提取公共接口
- ✅ 直接注入具体实现
- ✅ 使用事件驱动解耦

**❌ 绝对禁止**:

- 使用@Lazy注解
- 使用ObjectProvider延迟注入
- 使用ApplicationContext.getBean()依赖查找
- 使用@PostConstruct延迟初始化

**详细说明**: [验证流程指南 - 循环依赖解决](_docs/验证流程指南.md#⚠️-循环依赖解决原则)

### Q4: 何时使用三层架构（接口-抽象基类-实现）？

**A**:
- ✅ 通用服务（缓存、队列、存储）
- ✅ 可能有多种实现的服务
- ✅ 需要统一处理流程的服务
- ❌ 简单业务逻辑
- ❌ 一次性代码

**详细说明**: [业务代码编写规范.md - 接口实现分离](业务代码编写规范.md#84-接口实现分离模式三层架构)
