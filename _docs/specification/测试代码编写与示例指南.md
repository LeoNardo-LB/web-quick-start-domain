## 🚨 重要约束（强制执行）

### 1. 测试代码位置约束

**❌ 禁止事项**：

- ❌ **禁止**在业务模块（domain、app、adapter、infrastructure）的`src/test/java`中放置测试代码
- ❌ **禁止**在业务模块的`pom.xml`中添加测试依赖（junit、mockito等）
- ❌ **禁止**业务模块依赖test模块

**✅ 正确做法**：

- ✅ **所有测试代码必须放在test模块中**（`test/src/test/java`）
- ✅ 测试模块（test）依赖所有业务模块
- ✅ 通过Maven依赖传递，test模块可以访问所有业务代码

**原因**：

- 避免循环依赖（业务模块 → test → start → 业务模块）
- 统一测试管理
- 清晰的模块职责分离

### 2. 测试模块依赖关系

```
test → start → adapter → app → infrastructure → domain
```

**说明**：

- test模块依赖start模块
- start模块依赖所有业务模块
- 通过传递依赖，test模块可以访问所有业务代码进行测试

## 一、项目测试结构

### 1.1 项目目录结构

```
web-quick-start-domain/
├── adapter/                          # 接口层模块
│   └── src/main/java/               # 只有源代码，无测试代码
├── app/                              # 应用层模块
│   └── src/main/java/               # 只有源代码，无测试代码
├── domain/                           # 领域层模块
│   └── src/main/java/               # 只有源代码，无测试代码
├── infrastructure/                   # 基础设施层模块
│   └── src/main/java/               # 只有源代码，无测试代码
├── start/                            # 启动模块
│   └── src/main/java/
└── test/                             # ⭐ 独立测试模块（所有测试代码都在这里）
    └── src/
        └── test/java
            ├─cases
            │  ├─integrationtest    # 集成用例存放处
            │  └─unittest           # 单测用例存放处
            ├─fixture
            └─support
```

**⚠️ 关键规则**：

1. **所有测试代码必须在test模块中**
2. **业务模块（domain/app/adapter/infrastructure）不应包含src/test目录**
3. **业务模块的pom.xml不应包含测试依赖**

### 1.2 DDL文件约束

```
start/src/main/resources/config/schema.sql  # 唯一DDL文件，只读
```

**DDL修改约束**：

- ❌ **禁止**在测试代码中修改 `schema.sql` 文件
- ❌ **禁止**在运行时动态创建/修改表结构
- ❌ **禁止**执行任何DDL语句（CREATE/ALTER/DROP TABLE）
- ✅ **允许**在测试配置中引用此文件
- 🔒 **所有DDL变更必须通过版本控制流程**
- ⚠️ **测试只能读取DDL，不能写入或修改**

## 二、包名映射规则

### 2.1 测试类命名规范

| 测试类型     | 命名格式       | 示例                                           |
|----------|------------|----------------------------------------------|
| **单元测试** | `XxxUTest` | `UserServiceUTest`, `StringUtilsUTest`       |
| **集成测试** | `XxxITest` | `UserControllerITest`, `UserRepositoryITest` |

> `UTest`代表`UnitTest`单元测试，`ITest`代表`IntegrationTest`集成测试

### 2.2 包名、测试类对应关系

单测/集测包名是他们自己的包名加上业务包名，例如：

- 业务包名：{groupId}.xxx
- 单测包名：cases.unittest.{groupId}.xxx
- 集测包名：cases.integrationtest.{groupId}.xxx

类名命名规则：

- 业务类名：XxxService
- 单测类名：XxxServiceUTest
- 集测类名：XxxServiceITest

### 2.3 测试命名规范

```java
// ✅ 好的命名
void testCreateOrder_success()

void testCreateOrder_whenCustomerIdInvalid_thenThrowException()

// ❌ 避免的命名
void test1()

void testOrder()
```

### 2.4 Given-When-Then模式

```java

@Test
@DisplayName("创建订单成功")
void testCreateOrder_success() {
    // Given: 准备测试数据
    String customerId = "CUST001";
    Money totalAmount = new Money(100.00, "CNY");

    // When: 执行被测试方法
    OrderAggr order = OrderAggr.create(customerId, totalAmount);

    // Then: 验证结果
    assertThat(order.getCustomerId()).isEqualTo(customerId);
    assertThat(order.getStatus()).isEqualTo(OrderStatus.CREATED);
}
```

## 三、单元测试规范（**不启动Spring上下文**）

### 3.1 测试金字塔

```
        /\
       /  \      E2E测试（少量）
      /----\     集成测试（适量）
     /------\    单元测试（大量）
    /__________\
```

- **单元测试**：最多，快速，隔离（纯Mock）
- **集成测试**：适量，真实环境（Spring + H2）
- **E2E测试**：最少，慢，端到端

### 3.2 ✅ 允许的操作

```java
package

{groupId}.domain.service;

import support.UnitTestBase;
import org.junit.jupiter.api.Test;
import org.mockito.InjectMocks;
import org.mockito.Mock;

class {业务}ServiceUTest extends

UnitTestBase {

    @Mock
    private {业务} Repository orderRepository;

    @Mock
    private PaymentService paymentService;

    @InjectMocks
    private {业务} Service orderService;

    @Test
    void createOrder_ValidOrder_ReturnsOrder () {
        // Arrange
        Order order = new Order("test@example.com");
        when(orderRepository.save(any(Order.class))).thenReturn(order);

        // Act
        Order result = orderService.createOrder(order);

        // Assert
        assertNotNull(result.getId());
        assertEquals("test@example.com", result.getEmail());
    }

    @Test
    void createOrder_DuplicateEmail_ThrowsException () {
        // Arrange
        Order order = new Order("existing@example.com");
        when(orderRepository.existsByEmail("existing@example.com")).thenReturn(true);

        // Act & Assert
        assertThrows(Duplicate {业务} Exception.class, () -> orderService.createOrder(order))
    }
}
```

### 3.3 ❌ 绝对禁止的操作

```java
// 禁止Spring注解
@SpringBootTest  // ❌ 绝对禁止
@Autowired       // ❌ 绝对禁止

// 禁止数据库操作
DataSource dataSource = ...;  // ❌ 绝对禁止
Connection conn = ...;        // ❌ 绝对禁止

// 禁止DDL修改
String alterTable = "ALTER TABLE order ADD COLUMN new_column VARCHAR(255)"; // ❌ 绝对禁止

// 禁止文件操作
Files.

write(Paths.get("../schema.sql"), "MODIFIED CONTENT".

getBytes()); // ❌ 绝对禁止

// 禁止使用@Disabled跳过需要集成环境的测试
@Disabled("需要集成测试环境")  // ❌ 绝对禁止：单元测试必须使用Mock，不能依赖集成环境
```

**禁止使用@Disabled的原因**：

- ✅ 单元测试的核心原则是使用Mock隔离依赖，不依赖外部环境
- ✅ 如果测试"需要集成环境"，说明测试设计错误，应该改用Mock
- ✅ 使用@Disabled掩盖测试设计问题，导致测试覆盖度虚假
- ✅ 真正需要集成环境的测试应该放在集成测试文件（*IntegrationTest）中

**正确做法**：

```java
// ❌ 错误：使用@Disabled跳过测试
@Test
@Disabled("需要集成测试环境")
// 绝对禁止
void hybridSearch_ReturnsResults() {
    // ...
}

// ✅ 正确：使用Mock测试
@Test
void hybridSearch_ReturnsResults() {
    // Arrange - 使用Mock模拟依赖
    when(searchClient.search(anyString(), anyString(), anyInt(), anyInt()))
            .thenReturn(mockResponse);

    // Act
    SearchResult result = searchService.hybridSearch(index, query, Document.class);

    // Assert - 验证结果
    assertThat(result.getTotalHits()).isEqualTo(3);
    verify(searchClient, times(1)).search(anyString(), anyString(), anyInt(), anyInt());
}
```

### 3.4 单元测试最佳实践

1. **使用Mock隔离依赖**
    - Repository使用Mock
    - 外部服务使用Mock
    - 不访问真实数据库

2. **覆盖所有分支**
    - 正常流程
    - 异常流程
    - 边界条件

3. **快速执行**
    - 不启动Spring上下文 - 执行时间应 < 100ms

### 4.1 ✅ 允许的操作

```java
import support.ITestBase;
import org.junit.jupiter.api.Test;

import org.springframework.boot.test.mock.mockito.MockBean;

@MockBean
```

### 4.3 集成测试最佳实践

#### ⚠️ 核心原则：必须使用DBUnit管理测试数据

**强制要求**：

- ✅ **所有测试数据必须通过DBUnit加载**
- ❌ 禁止在测试代码中直接调用AppService/Repository创建数据
- ❌ 禁止使用HTTP POST请求在@BeforeEach中创建测试数据
- ❌ 禁止在测试方法中编写数据准备逻辑

**原因**：

1. **数据一致性**：DBUnit使用XML文件定义测试数据，数据结构清晰可见
2. **测试隔离**：CLEAN_INSERT确保每个测试方法的数据独立
3. **可维护性**：测试数据与测试代码分离，易于维护
4. **版本控制**：XML数据集文件可以纳入版本控制，数据变更可追溯

**正确示例**：

```java
class {业务}ControllerITest extends

ITestBase {

    @Override
    protected String getDataSetFile () {
        return "controller/order-data.xml"; // ✅ 使用DBUnit数据集
    }

    @Test
    void get {业务} ById_Existing {业务} _Returns200() throws Exception {
        // 测试逻辑，数据已在@BeforeEach中通过DBUnit加载
        mockMvc.perform(get("/api/orders/1"))
                .andExpect(status().isOk());
    }
}
```

**错误示例**：

```java

@BeforeEach
void setUpTestData() {
    // ❌ 禁止：直接调用Service创建数据
    orderAppService.createOrder(request);

    // ❌ 禁止：使用HTTP POST创建数据
    mockMvc.perform(post("/api/orders").content(json));

    // ❌ 禁止：直接操作数据库
    jdbcTemplate.execute("INSERT INTO order VALUES (...)");
}
```

#### 其他最佳实践

1. **Mock外部服务**
    - 使用@MockBean Mock外部RPC服务
    - 不Mock数据库

2. **测试完整流程**
    - Controller → Application → Domain → Repository
    - 验证HTTP请求/响应

3. **数据集文件组织**
    - 按功能模块组织：`datasets/integration/controller/`、`datasets/integration/service/`
    - 每个测试类对应独立的数据集文件
    - 数据集文件命名：{类名}-data.xml

## 七、各层测试示例

### Controller测试示例

#### 测试目标

验证Controller层：

- 请求参数验证
- 返回值格式
- 异常处理
- HTTP状态码

#### 完整示例

```java
package org.smm.archetype.adapter._example.order.web.api;

import com.fasterxml.jackson.databind.ObjectMapper;
import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.smm.archetype.app._example.OrderAppService;
import org.smm.archetype.domain._example.order.model.aggregateroot.OrderAggr;
import org.smm.archetype.domain.bizshared.base.Response;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.autoconfigure.web.servlet.WebMvcTest;
import org.springframework.boot.test.mock.mockito.MockBean;
import org.springframework.http.MediaType;
import org.springframework.test.web.servlet.MockMvc;

import static org.mockito.ArgumentMatchers.any;
import static org.mockito.Mockito.when;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.get;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.post;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.jsonPath;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.status;

/**
 * 订单Controller单元测试
 */
@WebMvcTest(OrderController.class)
@DisplayName("订单API单元测试")
class OrderControllerTest extends UnitTestBase {

    @Autowired
    private MockMvc mockMvc;

    @Autowired
    private ObjectMapper objectMapper;

    @MockBean
    private OrderAppService orderAppService;

    @Test
    @DisplayName("查询客户订单列表 - 成功")
    void testGetOrdersByCustomer_success() throws Exception {
        // Given: 准备Mock数据
        String customerId = "CUST001";
        OrderAggr order1 = OrderAggr.create(customerId, new Money(100.00, "CNY"));
        OrderAggr order2 = OrderAggr.create(customerId, new Money(200.00, "CNY"));

        when(orderAppService.getOrdersByCustomer(customerId))
                .thenReturn(List.of(order1, order2));

        // When & Then: 执行请求并验证
        mockMvc.perform(get("/api/orders")
                                .param("customerId", customerId)
                                .contentType(MediaType.APPLICATION_JSON))
                .andExpect(status().isOk())
                .andExpect(jsonPath("$.code").value(200))
                .andExpect(jsonPath("$.message").value("success"))
                .andExpect(jsonPath("$.data").isArray())
                .andExpect(jsonPath("$.data.length()").value(2))
                .andExpect(jsonPath("$.data[0].customerId").value(customerId));
    }

    @Test
    @DisplayName("创建订单 - 成功")
    void testCreateOrder_success() throws Exception {
        // Given
        String requestBody = """
                {
```

## 九、测试最佳实践

### DO ✅

1. **独立测试**：每个测试独立运行，互不依赖
2. **快速测试**：单元测试<100ms，集成测试<5s
3. **清晰命名**：`test{Method}_{Scenario}_{ExpectedResult}`
4. **覆盖边界**：正常、边界、异常情况都要测试
5. **使用断言**：assertThat > assertTrue

### DON'T ❌

1. **测试私有方法**：通过公共接口测试
2. **测试第三方库**：信任Spring、MyBatis等
3. **过度Mock**：只Mock外部依赖
4. **忽略测试**：避免@Disabled或@Ignore
5. **复杂测试逻辑**：测试代码应该简单清晰

## 十二、关键约束总结

### 12.1 目录与包名约束

- **测试模块**：所有测试代码必须在各模块的 `src/test/java/` 中
- **包名前缀**：无强制cases前缀，使用实际业务包名
- **类名后缀**：单元测试用 `XxxUTest`，集成测试用 `XxxITest`
- **基类位置**：`support.UnitTestBase.java` 和 `support.IntegrationTestBase.java`

### 12.2 DDL安全约束

- **文件位置**：`start/src/main/resources/config/schema.sql`
- **访问模式**：只读，禁止任何写操作
- **SQL约束**：禁止在测试中执行任何DDL语句
- **路径引用**：配置文件中使用 `classpath:config/schema.sql`

### 12.3 DBUnit数据管理约束

- **数据加载时机**：`@BeforeEach` + `@Transactional`
- **操作类型**：使用 `CLEAN_INSERT` 保证数据隔离
- **状态验证**：使用 `verifyDatabaseState()` 验证数据状态
- **文件命名**：数据集文件按功能模块组织，避免共享数据集

### 12.4 环境隔离约束

- **数据库隔离**：每个测试方法使用独立的H2内存数据库实例
- **文件隔离**：测试不能修改任何项目文件，特别是schema.sql文件
- **状态隔离**：每个测试方法执行前后，环境状态必须完全一致
- **Mock隔离**：单元测试Mock所有依赖，集成测试只Mock外部服务
