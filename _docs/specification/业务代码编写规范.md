### 1.3 Lombok使用规范

**基本原则**: 精确控制代码生成，避免注解冲突

#### 1.3.1 注解使用指南

| 注解                         | 使用场景       | 注意事项                           |
|----------------------------|------------|--------------------------------|
| `@Data`                    | ❌ **禁止使用** | 生成代码不可控，包含equals/hashCode可能不适用 |
| `@Getter`                  | ✅ 允许       | 仅生成getter                      |
| `@Setter`                  | ✅ 允许       | 仅生成setter                      |
| `@Builder`                 | ✅ 推荐       | 构建复杂对象                         |
| `@SuperBuilder`            | ✅ 推荐       | 继承场景                           |
| `@RequiredArgsConstructor` | ✅ 推荐       | 依赖注入                           |
| `@AllArgsConstructor`      | ⚠️ 谨慎      | 可能与@Builder冲突                  |
| `@NoArgsConstructor`       | ⚠️ 谨慎      | 可能与@RequiredArgsConstructor冲突  |
| `@Slf4j`                   | ✅ 推荐       | 日志输出                           |

#### 1.3.2 推荐模式

```java
// ✅ 推荐：精确控制
@Getter
@Setter
@Builder(setterPrefix = "set")
public class XxxDTO {

    private String id;
    private String name;

}

// ✅ 推荐：依赖注入
@RequiredArgsConstructor
public class XxxServiceImpl implements XxxService {

    private final XxxRepository xxxRepository;

}

// ❌ 禁止：全包注解
@Data  // 禁止！
public class XxxDTO {

    private String id;

}
```

### 1.4 日志规范

#### 1.4.1 日志级别

- **ERROR**: 错误，需要立即处理
- **WARN**: 警告，需要关注
- **INFO**: 关键业务流程
- **DEBUG**: 调试信息（生产环境关闭）

#### 1.4.2 日志使用

```java

@Slf4j
public class XxxService {

    public void process(XxxEntity entity) {
        log.info("Processing entity: id={}", entity.getId());
        try {
            // 业务逻辑
            log.debug("Entity processed successfully: id={}", entity.getId());
        } catch (Exception e) {
            log.error("Failed to process entity: id={}", entity.getId(), e);
            throw e;
        }
    }

}
```

**日志格式要求**:

- 使用占位符 `{}` 而非字符串拼接
- 异常日志必须包含异常对象
- 关键业务节点必须记录INFO日志

### 🎯 配置类命名规范（新增）

**核心规则：按聚合根命名，而非按层命名**

| 聚合根      | ✅ 正确配置类名                 | ❌ 错误配置类名（禁止使用）                                                                          |
|----------|--------------------------|-----------------------------------------------------------------------------------------|
| {业务}Aggr | `{业务}Configure.java`     | `{业务}AppConfigure.java` <br> `{业务}InfraConfigure.java` <br> `{业务}AdapterConfigure.java` |
| FileAggr | `FileConfigure.java`     | `FileAppConfigure.java` <br> `FileInfraConfigure.java`                                  |
| EventLog | `EventLogConfigure.java` | `EventLogConfig.java`（不能用Config后缀）                                                      |

**命名模式说明**：

```java
// ✅ 正确：按聚合根命名
@Configuration
public class {业务}Configure {}      // {业务}Aggr → {业务}Configure

@Configuration
public class FileConfigure {}       // FileAggr → FileConfigure

// ❌ 错误：按层命名（严格禁止）
@Configuration
public class {业务}InfraConfigure {} // ❌ 禁止！

@Configuration
public class {业务}AdapterConfigure {} // ❌ 禁止！

@Configuration
public class {业务}AppConfigure {} // ❌ 禁止！

// ❌ 错误：使用Config后缀（应为Configure）
@Configuration
public class {业务}Config {} // ❌ 禁止！必须用Configure
```

**设计原则**：

- ✅ 一个聚合根对应一个配置类
- ✅ 配置类按业务上下文聚合（如{业务}Aggr的所有Bean在{业务}Configure中）
- ✅ 使用 `Configure` 后缀（不是 `Config`）
- ❌ 不使用层次前缀（`Infra`、`Adapter`、`App`、`Domain`）

**例外情况**：

- 通用基础设施配置不受限（如 `EventConfigure`、`CacheConfigure`、`ThreadPoolConfigure`）
- 横切关注点配置不受限（如 `LogConfigure`、`NotificationConfigure`）

### 📋 Bean注册方式规范

┌─────────────────────────────────────────────────────────────────┐
│ 🚨 禁止使用的注解（严格禁止） │
├─────────────────────────────────────────────────────────────────┤
│ ❌ @Service - 业务服务层注解 │
│ ❌ @Component - 通用组件注解 │
│ ❌ @Repository - 数据访问层注解 │
│ ❌ @Controller - 控制器注解（仅adapter层Web层可用） │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│ ✅ 必须使用的方式 │
├─────────────────────────────────────────────────────────────────┤
│ 1. 创建配置类：@Configuration │
│ 2. 注册Bean：@Bean │
│ 3. 位置：{start}:src/main/java/{groupId}/config/ │

**正确示例**：

```java
// ✅ 正确：{start}:src/main/java/{groupId}/config/{业务}Configure.java
package {groupId}.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

/**
 * 订单聚合配置
 *
 * <p>聚合根：{业务}Aggr
 * <p>包含所有订单相关的Bean：应用服务、仓储、领域服务、事件处理器
 *
 * @author Leonardo
 * @since 2026/1/11
 */
@Configuration
public class {业务}Configure {

    @Bean
    public {业务}AppService orderApplicationService(
            {业务}AggrRepository orderRepository,
            {业务}DomainService orderDomainService){
        return new {业务}AppService(orderRepository, orderDomainService);
    }

    @Bean
    public {业务}AggrRepository orderAggrRepository({业务}AggrMapper orderAggrMapper){
        return new {业务}AggrRepositoryImpl(orderAggrMapper);
    }

    @Bean
    public {业务}DomainService orderDomainService(InventoryService inventoryService){
        return new {业务}DomainService(inventoryService);
    }

}
```

#### 2.1.2 依赖注入规范

**核心原则：优先使用构造器注入，特殊情况允许使用@Autowired并添加注释说明**

---

##### 📋 依赖注入规范速查表

| 场景                            | 注入方式                         | 要求                                    |
|-------------------------------|------------------------------|---------------------------------------|
| **业务代码**（Service、Repository等） | 构造器注入                        | ✅ 强制使用 `@RequiredArgsConstructor`     |
| **测试代码**                      | `@Autowired` 字段注入            | ✅ 允许使用                                |
| **配置类 - 跨配置类依赖**              | 构造器注入                        | ✅ 强制使用 `@RequiredArgsConstructor`     |
| **配置类 - 同配置类Bean依赖**          | `@Bean` 方法参数注入               | ✅ 推荐，可使用 `@Autowired(required=false)` |
| **配置类 - 条件装配特殊场景**            | `@Autowired(required=false)` | ✅ 允许，**必须添加注释说明**                     |

---

##### ⚠️ 严格禁止的场景（所有代码）

以下场景在任何代码中都**严格禁止**使用 `@Autowired`：

| 场景             | ❌ 禁止方式                                   | 原因           |
|----------------|------------------------------------------|--------------|
| 业务代码字段注入       | `@Autowired private XxxService service;` | 违反不可变性，难以测试  |
| 业务代码 Setter 注入 | `@Autowired public void setXxx(Xxx xxx)` | 破坏不可变性，依赖不明确 |
| 配置类跨配置类依赖      | `@Autowired private XxxConfig config`    | 应使用构造器注入     |

---

##### ✅ 允许使用 @Autowired 的场景

###### **场景1：测试代码（无条件允许）**

**测试代码**（`test/src/test/java` 下的所有代码）允许使用 `@Autowired` 字段注入：

```java
// ✅ 允许：测试类使用@Autowired
@SpringBootTest class {业务}

ControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Autowired
    private {业务} Repository orderRepository;

    @Test
    void testGetOrder() {
        // 测试逻辑
    }
}
```

**原因**：

- 测试代码不需要考虑不可变性
- `@Autowired` 字段注入简洁明了
- Spring 测试框架支持良好

---

###### **场景2：配置类中的条件装配（允许但需注释）**

在配置类的 `@Bean` 方法参数中，对于**条件装配的可选依赖**，允许使用 `@Autowired(required=false)`：

```java
// ✅ 允许：条件装配场景（必须添加注释说明）
@Configuration
public class EventConfigure {

    /**
     * 异步事件发布器
     *
     * <p>包装具体的事件发布器，提供异步发布能力。
     *
     * <p>💡 条件装配说明：
     * <ul>
     *   <li>KafkaEventPublisher和SpringEventPublisher是互斥的，通过条件装配只创建其中一个</li>
     *   <li>使用@Autowired(required=false)标记可选依赖，Spring会自动注入null如果Bean不存在</li>
     *   <li>运行时根据哪个Bean存在选择作为被包装对象</li>
     * </ul>
     *
     * <p>⚠️ 为什么要使用@Autowired(required=false)：
     * <pre>
     * 如果不使用@Autowired(required=false)，Spring会认为这两个参数都是必需的，
     * 即使Bean不存在也会尝试注入，导致启动失败。
     * 使用@Autowired(required=false)后，Spring会在Bean不存在时注入null，
     * 然后在方法体中通过null检查选择可用的实现。
     * </pre>
     *
     * @param kafkaEventPublisher  Kafka事件发布器（可选，条件装配）
     * @param springEventPublisher Spring事件发布器（可选，条件装配）
     * @return 异步事件发布器
     */
    @Bean
    public AsyncEventPublisher asyncEventPublisher(
            @Autowired(required = false) KafkaEventPublisher kafkaEventPublisher,
            @Autowired(required = false) SpringEventPublisher springEventPublisher) {

        // 根据哪个Bean存在，选择作为被包装对象
        // Spring的条件装配确保只有一个不为null
        EventPublisher delegate = (kafkaEventPublisher != null) ? kafkaEventPublisher : springEventPublisher;

        return new AsyncEventPublisher(delegate);
    }

}
```

**注释模板**（必须包含以下要素）：

```java
/**
 * Bean方法
 *
 * <p>💡 条件装配说明：
 * <ul>
 *   <li>说明为什么这两个依赖是可选的</li>
 *   <li>说明条件装配的条件（@ConditionalOnProperty等）</li>
 *   <li>说明运行时的选择逻辑</li>
 * </ul>
 *
 * <p>⚠️ 为什么要使用@Autowired(required=false)：
 * <pre>
 * 解释不使用@Autowired会发生什么问题（如启动失败）
 * 解释使用@Autowired后如何解决这些问题
 * </pre>
 *
 * @param xxx 可选参数说明
 * @return Bean说明
 */
```

##### 📊 规范对比总结

| 场景                 | 是否允许@Autowired | 是否需要注释   | 示例                                        |
|--------------------|----------------|----------|-------------------------------------------|
| **测试代码**           | ✅ 允许           | ❌ 不需要    | `@Autowired private MockMvc mockMvc;`     |
| **配置类 - 条件装配**     | ✅ 允许           | ✅ **必须** | `@Autowired(required=false) XxxBean bean` |
| **配置类 - 普通Bean依赖** | ❌ 禁止           | -        | 应使用构造器注入                                  |
| **业务代码**           | ❌ 禁止           | -        | 应使用@RequiredArgsConstructor               |

```java

@Configuration
@RequiredArgsConstructor  // 或者手动编写构造函数
public class XxxConfig {

    // 必需依赖
    private final DataSource dataSource;

    // 可选依赖 - 使用Optional包装
    private final Optional<KafkaEventPublisher> kafkaPublisher;

    @Bean
    public SomeBean someBean() {
        // 使用注入的依赖
        kafkaPublisher.ifPresent(publisher -> {
            // 使用publisher
        });
        return new SomeBean(dataSource);
    }

}
```

**同一配置类内的Bean依赖 - 使用@Bean方法参数注入**

- **✅ 推荐方式**：在 `@Bean` 方法参数上直接声明依赖，Spring会自动注入
- **✅ 可选依赖**：使用 `@Autowired(required = false)` 标记可选参数
- **💡 原因**：避免循环依赖，因为同一配置类内的Bean可能正在创建中

```java

@Configuration
public class XxxConfig {

    // Bean A
    @Bean
    public EventPublisher kafkaEventPublisher(...) {
        return new KafkaEventPublisher(...)
    }

    // Bean B 依赖 Bean A - 使用方法参数注入，避免循环依赖
    @Bean
    public AsyncEventPublisher asyncEventPublisher(
            @Autowired(required = false) KafkaEventPublisher kafkaPublisher,
            @Autowired(required = false) SpringEventPublisher springPublisher) {
        EventPublisher delegate = (kafkaPublisher != null) ? kafkaPublisher : springPublisher;
        return new AsyncEventPublisher(delegate);
    }

}
```

**循环依赖解决原则**

- **✅ 正确方式**：通过重构代码、解耦依赖、改进@Bean装配来解决
    - 跨配置类：使用构造器注入 + Optional
    - 同配置类：使用@Bean方法参数注入
- **❌ 绝对禁止**：
    - 使用@Lazy注解
    - 使用ObjectProvider延迟注入
    - 使用ApplicationContext.getBean()依赖查找
    - 使用@PostConstruct延迟初始化

#### 2.1.3 配置与条件化

- **配置绑定**：所有 `@ConfigurationProperties` 配置类必须通过 `@Import` 显式导入
- **条件装配**：必须使用 `@Conditional`、`@ConditionalOnProperty`、`@ConditionalOnClass` 等注解实现条件化 Bean 注入

**设计哲学**：通过显式配置提升代码可读性、可维护性和可测试性，降低框架魔法带来的认知负担。

### 2.2 Maven依赖管理规范

**核心原则**: 所有依赖的版本必须统一在根POM的`<dependencyManagement>`中管理，子模块引用依赖时不得显式声明版本号

> **⭐ 特别强调**：版本统一管理是Maven多模块项目的**基础规范**，违反此规范会导致：
> - 依赖版本不一致（不同模块使用不同版本）
> - 依赖冲突（传递依赖导致的版本冲突）
> - 升级困难（需要修改多个模块的版本号）
> - 维护成本高（分散的版本定义）

#### 2.2.1 依赖管理架构

```
┌─────────────────────────────────────────────────────────┐
│                  根POM (pom.xml)                        │
│  <dependencyManagement>                                 │
│   ├─ 系统模块依赖（domain、infrastructure等）            │
│   ├─ 第三方库依赖（Guava、Hutool、MyBatis-Flex等）      │
│   ├─ Spring Boot管理依赖（显式声明版本）                 │
│   └─ 依赖冲突解决（objenesis、byte-buddy等）            │
└─────────────────────────────────────────────────────────┘
                    ↓ 依赖版本管理
┌─────────────────────────────────────────────────────────┐
│               子模块POM (domain/pom.xml等)               │
│  <dependencies>                                         │
│   ├─ 只声明groupId和artifactId                          │
│   ├─ 禁止声明version                                    │
│   └─ 版本由根POM统一管理                                 │
└─────────────────────────────────────────────────────────┘
```

#### 2.2.2 根POM依赖管理规范

**必须在根POM的`<dependencyManagement>`中管理的依赖类型**:

1. **系统内模块依赖**:
   ```xml
   <dependency>
       <groupId>${project.groupId}</groupId>
       <artifactId>domain</artifactId>
       <version>${project.version}</version>
   </dependency>
   ```

2. **第三方库依赖**（非Spring Boot管理）:
   ```xml
   <dependency>
       <groupId>com.google.guava</groupId>
       <artifactId>guava</artifactId>
       <version>33.5.0-jre</version>
   </dependency>
   ```

3. **Spring Boot管理依赖**（显式声明版本以增强可控性）:
   ```xml
   <dependency>
       <groupId>org.springframework.boot</groupId>
       <artifactId>spring-boot-starter-web</artifactId>
       <version>3.5.9</version>
   </dependency>
   ```

4. **依赖冲突解决**:
   ```xml
   <!-- 优先使用较新版本 -->
   <dependency>
       <groupId>org.objenesis</groupId>
       <artifactId>objenesis</artifactId>
       <version>3.4</version>
   </dependency>
   ```

#### 2.2.3 子模块依赖声明规范

**✅ 正确示例**（不声明版本）:

#### 2.2.6 注解处理器（Annotation Processor）配置规范 ⭐ **重要**

**🚨 核心原则**：所有注解处理器必须在**根POM**中统一配置，子模块严禁重复配置 `maven-compiler-plugin` 的 `annotationProcessorPaths`

**适用范围**：

- MapStruct（对象转换）
- Lombok（代码生成）
- MyBatis-Flex（代码生成）
- Spring Boot Configuration Processor（配置元数据）
- 其他编译期代码生成工具

**⚠️ 常见错误**（绝对禁止）：

```xml
<!-- ❌ 错误：在子模块中重复配置注解处理器 -->
<build>
    <plugins>
        <plugin>
            <artifactId>maven-compiler-plugin</artifactId>
            <configuration>
                <annotationProcessorPaths>
                    <path>
                        <groupId>org.mapstruct</groupId>
                        <artifactId>mapstruct-processor</artifactId>
                        <version>${mapstruct.version}</version>
                    </path>
                </annotationProcessorPaths>
            </configuration>
        </plugin>
    </plugins>
</build>
```

**✅ 正确做法**（分两步）：

**步骤 1**：在根POM的 `<properties>` 中定义版本号：

```xml
<!-- 根POM pom.xml -->
<properties>
    <mapstruct.version>1.5.5.Final</mapstruct.version>
    <lombok-mapstruct-binding.version>0.2.0</lombok-mapstruct-binding.version>
</properties>
```

**步骤 2**：在根POM的 `<build><plugins>` 中统一配置注解处理器：

```xml
<!-- 根POM pom.xml -->
<build>
    <plugins>
        <plugin>
            <artifactId>maven-compiler-plugin</artifactId>
            <configuration>
                <annotationProcessorPaths>
                    <!-- Lombok -->
                    <path>
                        <groupId>org.projectlombok</groupId>
                        <artifactId>lombok</artifactId>
                        <version>${lombok.version}</version>
                    </path>
                    <!-- Spring Boot Configuration Processor -->
                    <path>
                        <groupId>org.springframework.boot</groupId>
                        <artifactId>spring-boot-configuration-processor</artifactId>
                    </path>
                    <!-- MyBatis-Flex Processor -->
                    <path>
                        <groupId>com.mybatis-flex</groupId>
                        <artifactId>mybatis-flex-processor</artifactId>
                        <version>${mybatis-flex.version}</version>
                    </path>
                    <!-- Lombok + MapStruct 绑定 -->
                    <path>
                        <groupId>org.projectlombok</groupId>
                        <artifactId>lombok-mapstruct-binding</artifactId>
                        <version>${lombok-mapstruct-binding.version}</version>
                    </path>
                    <!-- MapStruct Processor -->
                    <path>
                        <groupId>org.mapstruct</groupId>
                        <artifactId>mapstruct-processor</artifactId>
                        <version>${mapstruct.version}</version>
                    </path>
                </annotationProcessorPaths>
            </configuration>
        </plugin>
    </plugins>
</build>
```

**步骤 3**：子模块只需添加依赖，无需配置注解处理器：

```xml
<!-- 子模块 infrastructure/pom.xml -->
<dependencies>
    <!-- MapStruct（版本由根POM管理）-->
    <dependency>
        <groupId>org.mapstruct</groupId>
        <artifactId>mapstruct</artifactId>
    </dependency>
</dependencies>

        <!-- ✅ 不需要 build 配置，继承自根POM -->
```

**设计原因**：

1. **⚠️ 避免配置冲突**：多个模块配置不同的注解处理器会导致编译冲突
2. **✅ 统一版本管理**：确保所有模块使用相同版本的注解处理器
3. **✅ 简化子模块**：子模块只需添加依赖即可使用
4. **✅ 升级便利**：升级注解处理器版本只需修改根POM
5. **✅ 性能优化**：注解处理器统一配置，编译时只加载一次

**验证清单**：

添加注解处理器依赖后必须验证：

- [ ] 根POM的 `<properties>` 中定义了版本号
- [ ] 根POM的 `<build><plugins>` 中配置了 `annotationProcessorPaths`
- [ ] 根POM的 `<dependencyManagement>` 中管理了依赖版本
- [ ] 子模块POM中只添加了 `<dependency>`，未配置 `<plugin>`
- [ ] 运行 `mvn clean compile` 成功生成代码
- [ ] `target/generated-sources/annotations` 目录下生成了预期的代码

**实际案例**：

| 项目       | 根POM配置 | 子模块配置     | 结果          |
|----------|--------|-----------|-------------|
| ❌ **错误** | 无      | 每个子模块重复配置 | 配置冲突、版本不一致  |
| ✅ **正确** | 统一配置   | 只添加依赖     | 编译成功、代码生成正常 |

## 3. 领域建模规范

### 3.1 枚举设计规范

**核心原则**: 所有枚举字段必须标准化，禁止魔法值

#### 3.1.1 枚举识别规则

**字段语义包含（但不限于）以下关键词时，必须使用枚举**:

| 关键词       | 示例字段           | 枚举示例               |
|-----------|----------------|--------------------|
| type      | `xxxType`      | `XxxTypeEnum`      |
| status    | `xxxStatus`    | `XxxStatusEnum`    |
| state     | `xxxState`     | `XxxStateEnum`     |
| source    | `dataSource`   | `DataSourceEnum`   |
| business  | `businessType` | `BusinessTypeEnum` |
| errorCode | `errorCode`    | `ErrorCodeEnum`    |
| level     | `logLevel`     | `LogLevelEnum`     |
| mode      | `xxxMode`      | `XxxModeEnum`      |

> 需要根据这个规则扩展，主动根据变量名语义及作用进行识别。

#### 3.1.2 枚举转换规则

枚举与外部系统（数据库、API、文件等）交互时，需要完成 `Enum ↔ String` 的转换。根据**枚举值与存储值的关系**，分为两种情况：

---

##### **情况一：普通转换**（枚举名与存储值一致）

**适用场景**：

- 数据库存储的是枚举的 name（如 "CREATED", "PAID"）
- API 返回的也是枚举的 name
- 不需要额外的业务含义

**转换方式**：

```java
// 反序列化：String → Enum
public static {业务}

Status fromString(String value) {
    if (value == null || value.trim().isEmpty()) {
        return DEFAULT; // 或抛出异常
    }
    try {
        return {业务} Status.valueOf(value.toUpperCase());
    } catch (IllegalArgumentException e) {
        log.warn("Invalid {业务}Status: {}, using default: DEFAULT", value);
        return DEFAULT;
    }
}

// 序列化：Enum → String
public String toStatusString() {
    return this.name();
}
```

**实际应用**（在 Converter 中）：

```java

@Component
public class {业务}

AggrConverter {

    public {业务} Aggr toEntity ({业务} AggrDO dataObject){
        return {业务} Aggr.builder()
                       .setStatus(dataObject.getStatus() != null
                               ? {业务}Status.valueOf(dataObject.getStatus())  // String → Enum
                               : null)
                       .build();
    }

    public {业务} AggrDO toDataObject ({业务} Aggr entity){
        return {业务} AggrDO.builder()
                       .status(entity.getStatus() != null
                               ? entity.getStatus().name()  // Enum → String
                               : null)
                       .build();
    }
}
```

---

##### **情况二：特殊转换**（枚举名与存储值不一致）

**适用场景**：

- 数据库存储的是业务 code（如 "image", "document"）
- API 需要返回业务含义（如 "图片", "文档"）
- 存储值不是枚举名，而是自定义字段

**转换方式**：

```java

@Getter
public enum ContentType {
    IMAGE("image", "图片"),
    DOCUMENT("document", "文档"),
    VIDEO("video", "视频"),
    UNKNOWN("unknown", "未知");

    private final String code;        // 存储到数据库的值
    private final String description; // 业务描述

    // 反序列化：Code → Enum
    public static ContentType fromCode(String code) {
        if (code == null || code.trim().isEmpty()) {
            return UNKNOWN;
        }
        for (ContentType type : values()) {
            if (type.code.equalsIgnoreCase(code)) {
                return type;
            }
        }
        return UNKNOWN;
    }

    // 序列化：Enum → Code
    public String toCode() {
        return this.code;
    }

    // 序列化：Enum → Description（可选）
    public String toDescription() {
        return this.description;
    }
}
```

**实际应用**（在 Converter 中）：

```java

@Component
public class FileMetaConverter {

    public FileMetadata toEntity(FileMetadataDO dataObject) {
        return FileMetadata.builder()
                       .setContentType(FileMetadata.ContentType.fromCode(
                               dataObject.getContentType()))  // String → Enum
                       .build();
    }

    public FileMetadataDO toDataObject(FileMetadata entity) {
        return FileMetadataDO.builder()
                       .setContentType(entity.getContentType() != null
                               ? entity.getContentType().toCode()  // Enum → String
                               : null)
                       .build();
    }

}
```

---

##### **对比总结**

| 对比项       | 普通转换                         | 特殊转换                           |
|-----------|------------------------------|--------------------------------|
| **存储值**   | 枚举 name（CREATED, PAID）       | 业务 code（image, document）       |
| **反序列化**  | `{业务}Status.valueOf(string)` | `ContentType.fromCode(string)` |
| **序列化**   | `status.name()`              | `contentType.toCode()`         |
| **自定义方法** | 不需要                          | 需要 `fromXxx()` 和 `toXxx()`     |
| **典型场景**  | 状态机、流程控制                     | 内容类型、业务分类                      |
| **示例**    | {业务}Status, PaymentStatus    | ContentType, BusinessType      |

---

##### **命名规范**

- **普通转换**：可以使用 `fromString()` / `toString()`，或直接在 Converter 中调用 `valueOf()` / `name()`
- **特殊转换**：必须使用语义化命名，如 `fromCode()` / `toCode()`，`fromMimeType()` / `toMimeType()`

```java
// ✅ 推荐的命名
public static ContentType fromMimeType(String mimeType) { ...}

public String toMimeType() {return this.code;}

// ✅ 推荐的命名
public static {业务}

Status fromString(String status) { ...}

public String toString() {return this.name();}

// ❌ 避免模糊命名
public static ContentType parse(String s) { ...}  // 不明确是 parse 什么

public String convert() { ...}                    // 不明确转换为什么
```

#### 3.1.3 枚举定义位置

> **核心原则**：枚举如果与业务类强关联，则应作为内部枚举；相关字段应使用枚举类型而非String

---

##### **内部枚举**

**适用场景**：

- 与某个业务类强关联、不可拆分（如FileMetadata.ContentType）
- 仅在该类内部使用，不作为领域契约暴露
- 使用简短名称（Status、Type、Usage、ContentType等）

**编码要求**：

1. **字段类型必须使用枚举**（禁止使用String）：
   ```java

private ContentType contentType; // ✅ 正确
private final String contentType; // ❌ 禁止

```

2. **提供转换方法**（根据枚举值与存储值的关系，参考 [3.1.2节 枚举转换规则](#312-枚举转换规则)）：
    - **普通情况**（枚举名与存储值一致）：无需自定义方法，直接使用 `valueOf()` / `name()`
    - **特殊情况**（枚举名与存储值不一致）：提供 `fromXxx()` 和 `toXxx()` 方法

   **示例 - 特殊转换**：
   ```java
   // ContentType 使用 code 存储（"image", "document"）
   public static ContentType fromMimeType(String mimeType) {
       if (mimeType == null || mimeType.trim().isEmpty()) {
           return UNKNOWN;
       }
       String lower = mimeType.toLowerCase();
       if (lower.startsWith("image/")) return IMAGE;
       if (lower.startsWith("video/")) return VIDEO;
       return UNKNOWN;
   }

   public String toMimeType() {
       return this.code;
   }
   ```

3. **提供便捷业务方法**：
   ```java
   public boolean isImage() {
       return contentType == ContentType.IMAGE;
   }

   public boolean isDocument() {
       return contentType == ContentType.DOCUMENT;
   }
   ```

**完整示例**：

```java
public class FileMetadata extends Entity {

    private ContentType contentType;  // 使用枚举类型

    public boolean isImage() {
        return contentType == ContentType.IMAGE;
    }

    @Getter
    public enum ContentType {
        IMAGE("image", "图片"),
        DOCUMENT("document", "文档"),
        VIDEO("video", "视频"),
        UNKNOWN("unknown", "未知");

        private final String code;
        private final String description;

        ContentType(String code, String description) {
            this.code = code;
            this.description = description;
        }

        // 特殊转换：根据MIME类型判断
        public static ContentType fromMimeType(String mimeType) {
            if (mimeType == null || mimeType.trim().isEmpty()) {
                return UNKNOWN;
            }
            String lower = mimeType.toLowerCase();
            if (lower.startsWith("image/"))
                return IMAGE;
            if (lower.startsWith("video/"))
                return VIDEO;
            return UNKNOWN;
        }

        // 特殊转换：序列化为code
        public String toMimeType() {
            return this.code;
        }
    }
}
```

##### **外部枚举**

**适用场景**：

- 多个类共享的通用概念（如{业务}Status、PaymentMethod）
- 作为领域契约暴露给其他聚合
- 使用详细名称（避免命名冲突）

**位置规范**：

```

├── {业务}Status.java

```

**示例**：

##### **持久化处理**

**数据存储**：

- 数据库字段类型：VARCHAR（存储code或name）
- DO对象字段：String类型（与数据库列一致）

**转换位置**：

- 在Infrastructure层的Converter中完成String↔Enum转换
- 领域层和DO层互不可见对方的类型表示

**转换方式**（参考 [3.1.2节 枚举转换规则](#312-枚举转换规则)）：

- **普通转换**：在 Converter 中直接调用 `valueOf()` / `name()`
- **特殊转换**：在 Converter 中调用枚举的 `fromXxx()` / `toXxx()` 方法

---

##### **禁止事项**

- ❌ **禁止使用String存储类型/状态**：
  ```java

private String status; // ❌ 禁止

private {业务}Status status; // ✅ 正确
private final ContentType contentType; // ✅ 正确

```

- ❌ **禁止在领域层使用String进行枚举比较**：
  ```java

  ```

- ❌ **禁止在DO层或Converter层使用魔法值**：
  ```java
  // ❌ 错误：硬编码状态值
  if ("CREATED".equals(status)) { ... }

  // ✅ 正确：使用枚举

  ```

### 3.2 聚合根设计

**特征**:

- 是聚合的入口点
- 维护聚合内部的一致性边界
- 拥有唯一标识

```java

@Getter
public class XxxAggregate {

    private final XxxAggregateId  aggregateId;
    private       List<XxxEntity> entities;
    private       XxxStatus       status;

    // 业务行为
    public void addEntity(XxxValueObject value) {
        // 业务规则验证
        if (status != XxxStatus.READY) {
            throw new IllegalStateException("Cannot add entity in current state");
        }
        entities.add(new XxxEntity(value));
    }

    public void complete() {
        if (entities.isEmpty()) {
            throw new IllegalStateException("Cannot complete empty aggregate");
        }
        this.status = XxxStatus.COMPLETED;
        // 发布领域事件
        registerEvent(new XxxCompletedEvent(aggregateId));
    }

}
```

### 4.4 Mapper使用限制

**核心原则**: Mapper 仅在 Infrastructure 层使用，其他层必须通过 Repository 封装访问数据。

#### 4.4.1 使用规则

| 层级                 | Mapper使用 | Repository使用 | 说明                       |
|--------------------|----------|--------------|--------------------------|
| **Adapter**        | ❌ 禁止     | ✅ 允许         | 通过 Repository 访问数据       |
| **Application**    | ❌ 禁止     | ✅ 允许         | 通过 Repository 访问数据       |
| **Domain**         | ❌ 禁止     | ✅ 允许         | 定义 Repository 接口         |
| **Infrastructure** | ✅ 允许     | ✅ 允许         | 实现 Repository，可使用 Mapper |

#### 4.4.2 正确示例

```java
// ✅ 正确：Infrastructure层的Repository实现
@RequiredArgsConstructor
public class EventPublishRepository {

    private final EventPublishMapper eventPublishMapper;

    public EventPublishDO findByEventId(String eventId) {
        return eventPublishMapper.selectOneByQuery(
                QueryWrapper.create().where("event_id = ?", eventId)
        );
    }

}

// ✅ 正确：Adapter层通过Repository访问数据
@RequiredArgsConstructor
public class EventRetrySchedulerImpl {

    private final EventPublishRepository eventPublishRepository; // ✅ 使用Repository

    private void processEvent(EventConsumeDO consumeDO) {
        EventPublishDO publishDO = eventPublishRepository.findByEventId(consumeDO.getEventId());
        // ...
    }

}
```

#### 4.4.3 错误示例

```java
// ❌ 错误：Adapter层直接使用Mapper
@RequiredArgsConstructor
public class EventRetrySchedulerImpl {

    private final EventPublishMapper eventPublishMapper; // ❌ 禁止！应使用Repository

    private void processEvent(EventConsumeDO consumeDO) {
        EventPublishDO publishDO = eventPublishMapper.selectOneByQuery(...)// ❌ 禁止！
        // ...
    }

}
```

#### 4.4.4 设计原因

1. **分层隔离**: Mapper 是 MyBatis 技术细节，不应泄露到上层
2. **便于测试**: Repository 接口便于 Mock，提高可测试性
3. **技术替换**: 未来替换 ORM 框架时不影响上层代码
4. **职责清晰**: Infrastructure 层负责数据访问技术实现

#### 4.4.5 迁移指南

如果现有代码中其他层使用了 Mapper：

**步骤1**: 在 Infrastructure 层创建 Repository


```java

```


## 5. 分层开发规范

### 5.1 接口层（Adapter）

**职责**:

- 接收外部请求
- 参数验证
- 调用应用服务
- 返回响应

```java

@RestController
@RequestMapping("/api/xxx")
@RequiredArgsConstructor
public class XxxController {

    private final XxxAppService appService;

    @PostMapping
    public Result<XxxVO> create(@RequestBody @Valid XxxCreateRequest request) {
        // 1. 参数验证（通过@Valid）
        // 2. 调用应用服务
        Xxx entity = appService.create(request);
        // 3. 转换为VO
        return Result.success(XxxVO.from(entity));
    }

}
```

**禁止事项**:

- ❌ 直接调用Repository
- ❌ 包含业务逻辑
- ❌ 直接返回DO/Entity

### 5.2 应用层（Application）

**职责**:

- 用例编排
- 事务管理
- DTO转换
- 调用领域服务

```java

@ApplicationService
@RequiredArgsConstructor
public class XxxAppService {

    private final XxxRepository     xxxRepository;
    private final RelatedRepository relatedRepository;
    private final EventPublisher domainEventPublisher;

    @Transactional(rollbackFor = Exception.class)
    public Xxx create(XxxCreateRequest request) {
        // 1. 验证关联实体存在
        RelatedEntity related = relatedRepository.findById(RelatedId.of(request.getRelatedId()))
                                        .orElseThrow(() -> new RelatedNotFoundException(request.getRelatedId()));

        // 2. 创建实体（领域逻辑）
        Xxx entity = Xxx.create(related, request.getItems());

        // 3. 保存聚合
        xxxRepository.save(entity);

        // 4. 发布事件
        domainEventPublisher.publish(entity.getEvents());

        return entity;
    }

}
```

**禁止事项**:

- ❌ 包含领域业务逻辑（应在领域层）
- ❌ 直接访问数据库（应通过Repository）
- ❌ 处理HTTP相关逻辑

### 5.3 领域层（Domain）

**职责**:

- 核心业务逻辑
- 业务规则验证
- 领域事件发布

**特点**:

- ✅ 纯净的业务逻辑
- ✅ 无外部依赖
- ✅ 可独立测试

### 5.4 基础设施层（Infrastructure）

**职责**:

- 数据持久化实现
- 外部服务集成
- 技术组件实现

**特点**:

- ✅ 不包含业务逻辑
- ✅ 可替换实现
- ✅ 依赖倒置（实现Domain定义的接口）

**对象转换规范**:

- **✅ 强制规范**：必须使用 **MapStruct 转换器**（XxxConverter）进行对象转换
- **命名规范**：
    - `XxxMapper` - 专用于 **MyBatis 数据访问层**（数据库 CRUD 操作）
  - `XxxConverter` - 专用于 **MapStruct 对象转换器**（领域对象 ↔ DO/DTO 转换）

**MapStruct转换器使用示例**:

```java
// 1. 定义MapStruct Converter接口
@Mapper(
        componentModel = "spring",
        imports = {OrderStatus.class, PaymentMethod.class}
)
public interface OrderAggrConverter {

    /**
     * 领域对象转DO（用于新增）
     */
    @Mapping(target = "totalAmount", source = "totalAmount.amount")
    @Mapping(target = "status", expression = "java(order.getStatus().name())")
    @Mapping(target = "paymentMethod", expression = "java(order.getPaymentMethod().name())")
    OrderAggrDO toDO(OrderAggr order);

    /**
     * 更新DO（用于修改）
     */
    @Mapping(target = "totalAmount", source = "totalAmount.amount")
    @Mapping(target = "status", expression = "java(order.getStatus().name())")
    @Mapping(target = "paymentMethod", expression = "java(order.getPaymentMethod().name())")
    void updateDO(OrderAggr order, @MappingTarget OrderAggrDO orderDO);

    /**
     * 自定义枚举转换方法（default方法）
     */
    default OrderStatus stringToOrderStatus(String status) {
        return status == null ? null : OrderStatus.valueOf(status);
    }

}

// 2. 使用Converter（自动注入，MapStruct自动生成@Component）
@RequiredArgsConstructor
public class OrderAggrRepositoryImpl {

    private final OrderAggrMapper    orderAggrMapper;   // MyBatis Mapper
    private final OrderAggrConverter orderAggrConverter;  // MapStruct Converter（自动注入）

    public Order save(Order order) {
        OrderAggrDO orderDO = orderAggrConverter.toDO(order);
        orderAggrMapper.insert(orderDO);
        return order;
    }

    public void update(Order order) {
        OrderAggrDO orderDO = orderAggrMapper.selectOneById(order.getId());
        orderAggrConverter.updateDO(order, orderDO);
        orderAggrMapper.update(orderDO);
    }

}
```

**MapStruct常用注解说明**:

| 注解                             | 说明                | 示例                                                                |
|--------------------------------|-------------------|-------------------------------------------------------------------|
| `@Mapper`                      | 标记接口为MapStruct转换器 | `@Mapper(componentModel = "spring")`                              |
| `@Mapping`                     | 定义字段映射规则          | `@Mapping(target = "totalAmount", source = "totalAmount.amount")` |
| `@InheritInverseConfiguration` | 继承反向映射配置          | `@InheritInverseConfiguration`                                    |
| `@MappingTarget`               | 标记更新方法的目标参数       | `void updateDO(Order order, @MappingTarget OrderDO orderDO)`      |
| `@BeanMapping`                 | 定义Bean级别的映射规则     | `@BeanMapping(ignoreByDefault = true)`                            |

**设计原因**:

1. **类型安全**：编译期生成代码，字段映射错误在编译期发现
2. **性能优异**：无反射开销，性能接近手动编写的代码
3. **代码简洁**：减少70%-80%的样板代码
4. **维护性高**：字段新增/修改时，MapStruct自动检测编译错误
5. **灵活性**：支持复杂映射（嵌套对象、集合、自定义转换）

**注意事项**:

- 使用 `@Mapper(componentModel = "spring")` 自动注册为Spring Bean

- 使用 `default` 方法实现自定义转换逻辑
- 使用 `@InheritInverseConfiguration` 继承反向映射配置
- 使用 `@MappingTarget` 标记更新方法的目标参数
- Converter接口放在 `infrastructure/.../converter/` 包下
- MapStruct会在编译时自动生成实现类（如 `OrderAggrConverterImpl`）

## 6. 设计原则

### 6.1 SOLID原则

| 原则           | 说明          | 示例                   |
|--------------|-------------|----------------------|
| **S** - 单一职责 | 一个类只负责一件事   | XxxService只负责Xxx相关逻辑 |
| **O** - 开闭原则 | 对扩展开放，对修改关闭 | 使用接口+实现              |
| **L** - 里氏替换 | 子类可替换父类     | 子类不违反父类契约            |
| **I** - 接口隔离 | 接口小而专注      | 拆分大接口                |
| **D** - 依赖倒置 | 依赖抽象而非具体    | 依赖接口而非实现类            |

### 6.2 接口实现分离

**原则**: 通用服务必须定义接口

```java
// ✅ 正确：接口+实现
public interface XxxService {

    Result execute(Request request);

}

public class XxxServiceImpl implements XxxService {

    @Override
    public Result execute(Request request) {
        // 实现
    }

}

// ❌ 错误：直接使用实现类
public class XxxService {  // 缺少接口

    public Result execute(Request request) {
        // 实现
    }

}
```

### 6.3 依赖注入原则

**使用构造函数注入**:

```java
// ✅ 正确：构造函数注入
@RequiredArgsConstructor  // Lombok自动生成构造函数
public class XxxServiceImpl implements XxxService {

    private final XxxRepository  xxxRepository;
    private final RelatedService relatedService;

}

// ❌ 错误：字段注入
public class XxxServiceImpl implements XxxService {

    @Autowired
    private XxxRepository xxxRepository;  // 不推荐

}

// ❌ 错误：Setter注入
public class XxxServiceImpl implements XxxService {

    @Autowired
    public void setXxxRepository(XxxRepository xxxRepository) {
        this.xxxRepository = xxxRepository;
    }

}
```

### 6.4 接口-实现分离模式（三层架构）

> **核心原则**: 所有通用服务必须遵循三层架构模式，预留多种实现的扩展能力

#### 6.4.1 三层架构模式

```
┌─────────────────────────────────────────────────┐
│  Layer 1: Domain Interface (领域层接口)          │
│  - 定义业务契约（技术无关）                          │
│  - 位置: domain/_shared/service 或 domain/xxx    │
└─────────────────────────────────────────────────┘
                    ↓ defines
┌─────────────────────────────────────────────────┐
│  Layer 2: Abstract Base (抽象基类)              │
│  - 实现通用流程模板（日志、异常处理）                   │
│  - 定义扩展点（doXxx方法）                        │
│  - 位置: infrastructure/_shared/service/base     │
└─────────────────────────────────────────────────┘
                    ↓ extends
┌─────────────────────────────────────────────────┐
│  Layer 3: Concrete Implementation (具体实现)    │
│  - 接入具体技术（Redis、Kafka等）                  │
│  - 条件装配（@ConditionalOnProperty）            │
│  - 位置: infrastructure/xxx/service/impl         │
└─────────────────────────────────────────────────┘
```

**何时使用三层架构**:

- ✅ 通用服务（缓存、队列、存储等）
- ✅ 可能有多种实现的服务
- ✅ 需要统一处理流程的服务（日志、异常处理等）
- ❌ 简单的一次性业务逻辑
- ❌ 纯数据对象（DTO、VO等）

#### 6.4.2 模板方法模式

**场景**: 服务有通用流程但技术实现不同

**示例**:

```java
// Layer 1: Domain Interface
public interface XxxService {

    Result execute(Request request);

}

// Layer 2: Abstract Base with Template Method
public abstract class AbstractXxxService implements XxxService {

    @Override
    public final Result execute(Request request) {
        // 1. 通用前置处理
        validate(request);
        Request processed = preprocess(request);

        // 2. 抽象扩展点（由子类实现）
        Result result = doExecute(processed);

        // 3. 通用后置处理
        postprocess(result);

        return result;
    }

    /**
     * 扩展点：实际执行逻辑（由子类实现）
     */
    protected abstract Result doExecute(Request request);

    protected void validate(Request request) { /* 通用校验 */ }

    protected Request preprocess(Request request) {return request;}

    protected void postprocess(Result result) { /* 通用后处理 */ }

}

// Layer 3: Concrete Implementation
public class RedisXxxServiceImpl extends AbstractXxxService {

    @Override
    protected Result doExecute(Request request) {
        // Redis specific implementation
    }

}
```

#### 6.4.3 命名规范

| 层级             | 接口命名         | 抽象类命名                | 实现类命名                 |
|----------------|--------------|----------------------|-----------------------|
| Domain         | `XxxService` | -                    | -                     |
| Infrastructure | -            | `AbstractXxxService` | `RedisXxxServiceImpl` |

**后缀规范**:

- 接口: 无后缀或`Service`
- 抽象类: `Abstract`前缀
- 实现类: 技术栈前缀 + `Impl`后缀
    - ✅ `RedisCacheServiceImpl`
    - ✅ `KafkaEventPublisher`
    - ❌ `CacheServiceRedis` (错误)

**命名策略**:

1. **单一实现**: 使用 `{Name}Impl`
    - 示例: `UserRepositoryImpl`
2. **多实现**: 使用 `{Technology}{Name}Impl` 或 `{Name}{Technology}Impl`
    - 示例: `RedisCacheServiceImpl`, `KafkaEventPublisher`
3. **技术无关**: 不在接口名中暴露技术
    - ✅ `EventPublisher` (接口)
    - ❌ `KafkaEventPublisher` (作为接口名)

## 7. 中间件接入规范

### 7.1 中间件接入概述

#### 核心原则

**依赖检测 + 自动优先级**：基于中间件依赖的存在性自动选择实现，无需显式配置类型。

#### 新架构规则

1. **优先级规则**：
   - 没有接入中间件 → 使用本地组件
   - 如果引入了中间件 → 使用中间件（@Primary优先级）
   - 通过 `@ConditionalOnBean` 检测Bean存在性自动决定
   - 移除 `middleware.xxx.type` 显式类型选择

2. **配置规范**：
   - **SpringBoot标准Starter使用自带配置前缀**：
     - `spring.elasticsearch.*`（Elasticsearch）
     - `spring.kafka.*`（Kafka）
     - `spring.data.redis.*`（Redis）
   - **middleware.* 前缀仅用于**：
     - 本地组件参数（如 `middleware.cache.local.initial-capacity`）
     - 非标准、无自配参数的中间件引入
     - 业务参数（如 `middleware.cache.redis.key-prefix`）

#### 实现模式

所有中间件配置遵循以下模式：

```java
// 中间件可用时（Spring Boot自动配置了Bean）
@Bean
@ConditionalOnBean(MiddlewareBean.class)  // 检测中间件Bean存在
@Primary                                   // 设置中间件为优先实现
public ServiceClient middlewareClient(MiddlewareBean bean) {
    return new MiddlewareClientImpl(bean);
}

// 中间件不可用时（使用本地组件）
@Bean
@ConditionalOnMissingBean(MiddlewareBean.class)  // 检测中间件Bean不存在
public ServiceClient localClient() {
    return new LocalClientImpl();
}
```

#### 设计优势

- **零配置**：无需手动配置类型，自动检测依赖
- **Spring Boot原生**：使用Spring Boot标准配置，遵循框架约定
- **易于扩展**：新增中间件只需实现接口，添加 `@ConditionalOnBean` 和 `@Primary`
- **清晰优先级**：通过 `@Primary` 注解明确优先级，避免歧义
- **技术无关**：Domain层不感知具体技术实现

> **💡 实现基础**: 所有中间件服务都遵循[第6.4节](#64-接口实现分离模式三层架构)定义的三层架构模式和模板方法模式

### 7.2 中间件映射表

| 中间件         | 本地组件           | 检测方式（@ConditionalOnBean）                                 | SpringBoot标准配置前缀               | middleware.*适用范围                     |
|------------|----------------|-----------------------------------------------------------|----------------------------------|-----------------------------------|
| **Redis**   | Caffeine        | `RedisTemplate.class`                                       | `spring.data.redis.*`             | 业务参数（key-prefix, default-ttl等）       |
| **Kafka**   | Spring Events   | `KafkaTemplate.class`                                       | `spring.kafka.*`                  | 业务参数（topic-prefix, enable-acks等）      |
| **Elasticsearch** | DisabledSearch | `ElasticsearchOperations.class`                              | `spring.elasticsearch.*`           | 无（使用SpringBoot标准配置）                 |
| **RustFS OSS** | LocalFileStorage| `RustFsOssClientImpl.class`                                 | 无（非标准中间件）                    | 完整配置（endpoint, access-key等）          |
| **阿里云短信** | -             | `AliyunSmsClient.class`（需要手动配置）                           | 无                               | 完整配置（provider, access-key-id等）      |
| **阿里云邮件** | -             | `AliyunEmailClient.class`（需要手动配置）                         | 无                               | 完整配置（provider, smtp-host等）         |

#### 检测方式说明

- **@ConditionalOnBean**：检测中间件提供的Bean是否存在
  - `RedisTemplate.class`：Spring Boot自动配置（需引入spring-boot-starter-data-redis）
  - `KafkaTemplate.class`：Spring Boot自动配置（需引入spring-boot-starter-kafka）
  - `ElasticsearchOperations.class`：Spring Boot自动配置（需引入spring-data-elasticsearch）
  - `RustFsOssClientImpl.class`：自定义Bean（需在OssConfigure中定义）

- **@Primary**：中间件Bean设置为优先实现
  - 当中间件和本地组件都存在时，优先使用中间件实现
  - 通过 `@ConditionalOnMissingBean` 确保两者互斥

#### 配置策略对比

| 配置项              | 旧架构（已废弃）            | 新架构（当前）                         |
|-------------------|-----------------------|----------------------------------|
| 类型选择              | `middleware.cache.type = "redis"` | 无需配置（自动检测RedisTemplate）        |
| SpringBoot标准配置      | 可选                    | **强制使用**（`spring.data.redis.*`）    |
| middleware.*业务参数  | 所有配置（含类型选择）          | **仅业务参数**（key-prefix, default-ttl） |
| 本地组件参数           | 混合在中间件配置中           | **独立配置**（middleware.cache.local.*） |

```java
import org.springframework.boot.autoconfigure.condition.ConditionalOnBean;
```

```java
@ConditionalOnBean(HazelcastInstance.class)
```

| 禁止项                        | 说明     | 正确做法                 |

| ❌ 硬编码配置值                   | 不易维护   | 使用配置文件               |
| ❌ 使用System.out.println     | 缺乏日志管理 | 使用@Slf4j             |
| ❌ 空catch块                  | 吞噬异常   | 至少记录日志               |
| ❌ 在循环中查询数据库                | N+1问题  | 批量查询                 |
| ❌ 大事务操作                    | 锁定时间长  | 拆分小事务                |
| ❌ 过度使用synchronized         | 性能问题   | 使用分布式锁               |

### 8.2 安全层面

| 禁止项         | 说明    | 正确做法           |
|-------------|-------|----------------|
| ❌ 日志记录敏感信息  | 泄露风险  | 脱敏处理           |
| ❌ 明文存储密码    | 安全风险  | BCrypt加密       |
| ❌ 客户端存储敏感凭证 | XSS风险 | 服务端Session     |
| ❌ 使用不安全随机数  | 可预测性  | 使用SecureRandom |
| ❌ SQL拼接     | SQL注入 | 使用参数化查询        |
| ❌ 直接反射用户输入  | RCE风险 | 白名单验证          |

### 8.3 性能层面

| 禁止项         | 说明       | 正确做法     |
|-------------|----------|----------|
| ❌ 循环中查数据库   | N+1问题    | 批量查询/缓存  |
| ❌ 不必要的对象创建  | GC压力     | 复用对象/对象池 |
| ❌ 过度锁粒度     | 并发性能降低   | 缩小锁范围    |
| ❌ 阻塞IO在虚拟线程 | 虚拟线程优势丧失 | 使用NIO    |
| ❌ 不分页查询大表   | OOM风险    | 强制分页     |

### 8.4 架构层面

| 禁止项         | 说明                                                        |
|-------------|-----------------------------------------------------------|
| ❌ 内层依赖外层    | 违反依赖倒置                                                    |
| ❌ 领域层依赖基础设施 | 领域层应保持纯净                                                  |
| ❌ 跨层访问      | 遵循分层调用链                                                   |
| ❌ 修改生成代码    | `{groupId}.infrastructure._shared.generated`是自动生成的包，禁止修改！ |

## 9. 测试规范

> **核心原则**：单元测试与集成测试严格分离，100%通过率，95%行覆盖率，100%分支覆盖率
>
> **详细规范**：本章节为测试规范概述，详细内容请参考 [测试代码编写与示例指南.md](测试代码编写与示例指南.md)

### 9.1 测试分层策略

| 测试类型     | 基类                  | Spring上下文 | 数据隔离           | 示例                            |
|----------|---------------------|-----------|----------------|-------------------------------|
| **单元测试** | UnitTestBase        | ❌ 不启动     | Mock           | {业务}ServiceTest               |
| **集成测试** | IntegrationTestBase | ✅ 启动      | @Transactional | {业务}ControllerIntegrationTest |
| **启动测试** | -                   | ✅ 完整启动    | -              | ApplicationStartupTests       |

**核心区别**：

- **单元测试**：测试单个类的逻辑，使用Mock隔离所有依赖，执行速度快
- **集成测试**：测试多层协同，使用真实Bean，Mock外部服务，验证集成点
- **启动测试**：验证Spring上下文能正常启动，无Bean装配错误

### 9.2 测试原则

#### 9.2.1 核心原则

- ✅ **测试分离**：单元测试不启动Spring，集成测试启动最小化上下文
- ✅ **100%通过率**：所有测试必须通过，不允许失败
- ✅ **高覆盖率**：行覆盖率≥95%，分支覆盖率100%
- ✅ **快速反馈**：单元测试执行<100ms，集成测试执行<500ms

#### 9.2.2 单元测试原则

- ✅ 使用 `UnitTestBase` 基类
- ✅ Mock所有依赖（Repository、外部服务等）
- ✅ 覆盖所有public方法和异常分支
- ✅ 不访问数据库、文件系统等外部资源

**详细规范**：[测试代码编写与示例指南.md - 单元测试](测试代码编写与示例指南.md#三单元测试规范不启动spring上下文)

#### 9.2.3 集成测试原则

- ✅ 使用 `IntegrationTestBase` 基类
- ✅ 启动最小化Spring上下文
- ✅ 使用DBUnit管理测试数据
- ✅ Mock外部服务，不Mock内部Bean
- ✅ 使用@Transactional保证数据隔离

**详细规范**：[测试代码编写与示例指南.md - 集成测试](测试代码编写与示例指南.md#四集成测试规范启动spring上下文)

#### 9.3.1 UnitTestBase（纯单元测试基类）

#### 9.3.2 IntegrationTestBase（集成测试基类）

- ✅ 启动完整Spring上下文
